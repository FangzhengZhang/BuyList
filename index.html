Introduction

The system our team designed is a trading system, a platform where the buyer can post what they want, and other users can see the post and try to connect with the buyer. The user can login and has own profile. Users can post a request about what they want with some descriptions, such as pick-up or ship, prices, zip code, e-mail address or phone number. Some of the descriptions are optional. Those posts will be shown on the platform. If someone could offer the request stuff, then he/she/them could connect with the buyer for more details. Other users could search by zip codes or keywords to obtain more specific posts. If the trade is completed or the buyer no longer needs it, the buyer can mark the post as done, then the post will disappear from the platform, but the buyer still can see all posts from history in profile. Every post has a one week time limitation, after one week the post will expire and disappear.

We are going to use a data central structure as users can require data which is stored on our server, and abstract data types style as the whole system will be built on object oriented languages. This system will have three main parts:

  • Front-end: implement with JavaScript, HTML, CSS. This section will be used for display buyer and seller information, and the used goods details.
  • Back-end: implement with Java, python, and shell script. The backend will handle user’s requests and load the file or data from the database and the server file system.
  • Database: we are planning to use MySQL and MongoDB as our primary database. The user and goods data will be stored in the database.

The idea of this project came from VT student group chat, where students do used goods trade. We found that people not only post used items in the chat, they also ask people whether they have something that he/she/them want. So we thought a buyer-lead used goods trade platform would be an interesting idea. This project is a completely new idea that we have never worked on before. So the progress of this project is zero at this point.

Requirements Summary

The requirements of this project are listed below:

  1. Buyers can create a profile and username/avatar for their account.
  2. Sellers can create a profile and username/avatar for their account.
  3. Buyers can post requests for what they want.
  4. Buyer requests can contain descriptions, pick-up or shipping information, target prices, zip codes, e-mail address and/or phone number.
  5. Some of the descriptions are optional.
  6. Requests will be visible through the BuyerList tool/application.
  7. Other people can see and review requests.
  8. Users can search requests by zip codes or key words to find specific requests.
  9. People can connect with the requester to make an offer to satisfy the request.
  10. Potential sellers can connect with the buyer through the application and exchange more details.
  11. If the trade is done, the request post is marked as completed.
  12. A buyer can change their mind and mark the request as “No longer needed”, which will complete the request and archive it.
  13. Every request post has a one week time limit. After one week, the request expires and is removed.
  14. Completed request posts are removed from the active database and moved to the archive.

High-Level Design

Functions/Modules:

The BuyerList tool has the following functions/modules:

  • Create New User
    • A user of the BuyerList system could be a buyer who is requesting something or a seller who is offering a requested item. In either case, a user needs to have a profile created in the system before they can use the system. The user’s profile will consist of a username, password and personal contact information such as their full name, e-mail address and phone number. Users will be identified within the BuyerList system by their username.
    • This satisfies requirement 01 and requirement 02.
  • User Logon
    • Users will logon to the BuyerList system by their username and password.
  • Create New Request
    • A user can create a new request for something they want. The request needs to contain the following information:
      • Title of what is being requested. This is a required entry.
      • A description of the request. This is a required entry.
      • Condition of the request. This is an optional entry. The user will be able to select from condition options: New, Like new, Excellent, Good, Fair, and Salvage.
      • Transportation information, such as requesting delivery to a location or willing to pick-up. This is a required entry.
      • Zip code where the requester lives or where delivery is being requested. This is a required entry.
      • The username of the requester (buyer) will automatically be assigned to the request.
      • The date and time of the request creation is automatically assigned to the request.
      • Automatically calculate and assign the date and time for the expiration of the request record. Requests have a one-week expiration.
    • This satisfies requirement 03, requirement 04, and requirement 05.
  • Search Requests
    • Enter a search criteria to see requests. Search criteria options are a combination of the following:
      • Title field keyword search.
      • Description field keyword search.
      • Condition field search.
      • Transportation field keyword search.
      • Zip code field value, value list, or value range search.
      • Offer description field keyword search.
      • Requester username field search by full name or partial name.
      • Seller username field search by full name or partial name.
    • Search results will be displayed as a list in table format.
    • This satisfies requirement 06, requirement 07, and requirement 08.
  • Select Request
    • Select a request from the search results list for further processing. This will open the single request record and display the contents of all the fields.
    • This satisfies requirement 05 and requirement 07.
  • Submit Offer
    • A seller can submit an offer to satisfy a request. The seller does this by selecting and opening the request they want to make an offer on. The seller then types the description of their offer to the buyer in the Offer Description field. The seller’s username is automatically added to the offer.
    • Each offer to satisfy a request should be automatically appended to the end of the request in an offer section.
    • This satisfies requirement 09 and requirement 10.
  • Accept Offer
    • The buyer who submitted the request can accept one of the offers to satisfy their request. This should be a checkbox in the offer section of the request.
  • Reject Offer
      The buyer who submitted the request can reject offers to satisfy their request. This should be a checkbox in the offer section of the request.
  • Request Not Needed
    • The buyer can update a request and mark it as “No Longer Needed”. This will automatically cause the request to be set to a completed status.
    • This satisfies requirement requirement 12.
  • Request Completed
    • Change the status of the request to “Completed”.
    • No more changes can be made to this request.
    • Have the system send a message to the buyer that this request is complete.
    • Have the system send a message to all the sellers who made an offer on this request that this request is complete.
    • This satisfies requirement requirement 11.
  • Request Expired
    • Each request has an one-week expiration period. If it is still open after one week, have the system automatically change the status to “Completed”.
    • This satisfies requirement requirement 13.
  • Archive Request
    • Move requests with a “Completed” status into the archive.
    • Requests which have been archived can only be searched by the buyer who created the request.
    • This satisfies requirement requirement 14.

System Functions:

The BuyerList tool has the following system functions:

  • Verify username is unique.
    • Every username in the BuyerList tool must be unique. When a new user is creating their user profile, the system will check whether the username they request is available. If the username already exists, then the system will throw an error that the username has been taken and a different username must be requested.
  • Verify e-mail address is valid.
    • Verify that the e-mail address provided by the user in their profile is a valid e-mail address and that the user can be connected through that e-mail address.
  • Verify phone number is valid.
    • Verify that the phone number provided by the user in their profile is a valid phone number and that the user can be connected through that phone number.
  • Assign date and time stamp to new requests when created.
    • Capture the date and time when a new request is created and assign it to the request. The date and time is used to calculate the expiration date and time for the request.
  • Assign expiration date and time to new request when created.
    • Calculate the expiration date and time for the request when it is created by using the creation date and time for the request. The calculated expiration date and time will be one week from the creation date and time.
  • Set request status to “Completed” when offer accepted.
    • After the requester (buyer) accepts an offer for the request, automatically change the status of the request to “completed”.
  • Set request status to “Completed” when request no longer needed.
    • After the requester (buyer) marks the request as “No longer needed”, automatically change the status of the request to “completed”.
  • Set request status to “Completed” when request expires.
    • When the expiration date and time for an open request is reached, automatically change the status of the request to “completed”.
  • Notify the buyer and all potential sellers when a request status changes to “Completed”.
    • When the status of a request is changed to “Completed”, then automatically send a message to the requester (buyer) who created the request and to any seller who has submitted an offer on the request. The message will inform them that the request is completed.
  • Move completed requests to the archive.
    • When the status of a request is changed to “Completed”, then automatically move the request into the archive section of the database.

State Changes:

The BuyerList tool has the following states and state changes for a request:

  • Request: Created
    • “Created” is the initial state for each request after it has been initially created.
    • From the “created” state, a request can be changed to the “offer received” state when an offer has been submitted against the request.
    • From the “created” state, a request can be changed to the “completed” state when the buyer marks the request as “no longer needed”.
    • From the “created” state, a request can be changed to the “completed” state when the expiration date and time is reached without an offer being accepted.
  • Request: Offer Received
    • The state of a request is changed to “offer received” when an offer has been submitted against the request.
    • From the “offer received” state, a request can be changed to the “offer accepted” state when the buyer accepts an offer submitted against their request.
    • From the “offer received” state, a request can be changed to the “completed” state when the buyer marks the request as “no longer needed”.
    • From the “offer received” state, a request can be changed to the “completed” state when the expiration date and time is reached without an offer being accepted.
  • Request: Offer Accepted
    • The state of a request is changed to “offer accepted” when an offer has been accepted by the buyer against their request.
    • Once the state of the request is set to “offer accepted”, the state of the request is automatically changed to “completed”.
  • Request: Completed
    • The state of a request can be changed to “completed” three possible ways.
      • The buyer can decide that their request is “no longer needed”, which will change the state of the request to “completed”.
      • The expiration date and time for the request can be reached, which will change the state of the request to “completed”.
      • The buyer can accept and offer against the request, which will change the state to “offer accepted” and then automatically change the state to “completed”.

Quality Attributes:

In order to achieve the design objectives of the BuyerList tool, many design decisions are needed which will impact the design of the software architecture. Helping to drive those design decisions are the quality attributes which we feel are the most important for the success of the system. Those quality attributes are:

  • Usability
  • Portability
  • Flexibility
  • Maintainability
  • Testability

Usability is the most important quality attribute for the BuyerList tool because it is entering a competitive service area. Out potential customers have a variety of choices in the used goods arena, so our tool must be very user-friendly, or our customers will use another option. If users do not see the BuerList tool as useful, then they will not use it.

Two tier client-server software architectures improve usability and scalability. Usability is increased though user-friendly, form-based interfaces. Scalability is improved because two tiered systems can hold up to 100 users, whereas file server architectures can only accommodate 12.

Three tier client server architecture is also known as multi-tier architecture and signals the introduction of a middle tier to mediate between clients and servers. The middle tier exists between the user interface on the client side and database management system (DBMS) on the server side. This third layer executes process management, which includes implementation of business logic and rules. The three tier models can accommodate hundreds of users.

Portability is an important quality attribute because we want our tool to be available to as many users as possible by making our tool available over multiple platforms. We want to serve customers who use their computer or their mobile device. We want to serve users who are on a PC, Android and Apple environment. This means, the BuyerList tool will need to be able to operate on the Windows, macOS, Android, and iOS operating systems. The goal would be to have a single solution which can be ported over to the different operating system.

The clients can be ported to many different operating systems and still use the same server. The clients can each be optimized to perform best for each specific operating system. Due to the multi-tiered architecture, the server will not have to change to accommodate the different clients.

Flexibility is an important quality attribute because in the competitive arena of used goods, the BuyerList tool will need to keep up with and adjust to the changing needs of our customers and the market. For the tool to be useful and remain useful to our potential customers, it must provide the features they want. As the needs of our customers change and evolve, the BuyerList tool must be able to quickly change and evolve with those needs. This means the software architecture must be flexible so that it can be modified to add or change the functionality of the tool.

Client-server software architecture is versatile and flexible in today’s fast-changing IT landscape. It is modular in structure and relies on messaging services for communication between components. They were designed to improve flexibility, usability, scalability, and interoperability. Software flexibility implies the ability for a program to change easily according to different users and different system requirements.

Maintainability is an important quality attribute of the tool because defects and errors must be addressed quickly. The BuyerList tool cannot risk a negative customer experience due to defects or functionality that does not work correctly. Defects need to be prevented from reaching the customer. But, if they do, must be fixed right away. The software architecture needs to enable rapid maintenance and correction of defects and errors.

Testability is an important quality attribute for the tool because it supports flexibility and maintainability. The BuyerList tool will need to change continuously due to either changing customer needs or maintenance fixes. Testability is essential for being able to verify that these changes work and do not introduce other issues. Most issues come from either a new function or a fix to an existing function. Testing will be critical to ensuring that changes to not introduce issues.

Design Rationale

The BuyerList tool will be using a combination of Object-Oriented Organization and Client-Server for its software architecture. The focus of the BuyerList are the requests created by the users who are buyers. Each request is independent from the other requests. There is no interaction between the different individual requests. There is even no interaction between requests from the same user. One request does not relate to another request. Because of this independence between request objects, the Object-Oriented Organization was the best choice for the software architecture for the BuyerList tool.

Basically, our application focus on dealing with users’ request data. The application doesn’t require a complex framework to reach the requirements. Modifying and displaying data from databases can satisfy all operations. Therefore, based on the Object-Oriented Organization, just several object components are required and those components only need to process the user’s request and store it into the database or obtain the specific data from the database. For the user’s posting requirements, the user will send the descriptions, details, and image files to the server. First, the components in the server will check whether the request contains all required information and the formats. If the request is qualified, the other component will create a new request data instance store all information of the request, and store it into the database. For the searching request, after receiving the qualified request, the component will search the target from the database, if the target is found, then the component will send the data back to the client and display it as HTML on the user’s browser. Otherwise, it will send a non-result alert.

Decision to over the BuyerList tool for computer and mobile users. For computer users, the tool will server both Windows and macOS users. For mobile users, the tool will server Android and iOS users.

  • Usability
    • Our application is user-friendly. There are no complex operations in usage, all operations in our application are intuitive and simple. Users don’t have to spend time learning how to use it.
  • Portability
    • Our application is a web-based and client-server model, users can access the application by using any browser on any device.
  • Maintainability
    • Our application only contains three parts: Client (HTML, Js), Server (Java) and Database(MySQL). There is a specific company that is responsible for maintaining the database. So we only need to maintain the Client and Java part.
  • Testability
    • Our use cases include almost all situations and our website is similar to other common shopping websites, so there are many testing examples we can use.
  • Flexibility
    • Our application is Object-Oriented, the influences among each object are limited. So, changing the objects is simple and wouldn’t cause any unpredictable error. The client-side is HTML-based. Therefore, making changes in the client is simple and direct.
